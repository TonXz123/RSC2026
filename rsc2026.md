2. Introduction
A testing organiser is running a regional web technologies competition in an isolated local area network (LAN) environment. Each competitor must build a complete full-stack web application that can be accessed from the internal room network only. Internet access is not available.
The competitor is responsible for ensuring that the submitted application URLs remain accessible within the local competition network.
Operational Constraints

Isolated LAN Deployment: The application operates inside a completely isolated LAN with NO internet connectivity.
Strict Restrictions: External APIs, external web fonts (e.g., Google Fonts), external CDNs (e.g., loading Tailwind or Bootstrap via CDN link), and cloud deployment are STRICTLY PROHIBITED. All assets, libraries, and dependencies must be bundled locally.
Database Architecture: Relational/SQL database (PostgreSQL preferred based on seed syntax). The application must seamlessly run with the provided seed data (users, candidates, session_status, submissions, results). The competitor is allowed to extend the schema if required, but existing tables and data relations MUST NOT be broken.


3. Scenario
You are asked to build a Test Submission Management System for a competition room.
The system must allow candidates to review the task, submit application URLs, and view submission status. Judges must be able to open and close the testing session, re-check submissions, and confirm results. Managers must be able to view score summaries and ranking dashboards.

All endpoints must be implemented exactly as defined below. Any route deviation or missing endpoint will result in an automatic 0 score for that sub-section.

Module Breakdown
Authentication Module

POST /api/login — Authenticates a user using credentials. Returns an access token/session.
POST /api/logout — Logs out the current user and invalidates the session token.

Candidate Module

GET /api/config — Returns the global current session status, system configuration, and timing data (countdown parameters).
GET /api/tasks — Returns the specific task overview and document summary for the authenticated user.
GET /api/my-submission — Fetches the active candidate submission details.
POST /api/my-submission — Creates a new candidate submission containing Frontend URL and Backend API URL.
PUT /api/my-submission — Updates the candidate's existing URLs. Allowed only before the testing session closes.
GET /api/my-result — Returns the latest automated grading result card for the currently logged-in candidate.

Judge Module

GET /api/candidates — Fetches the global list of all candidates for the judge dashboard.
GET /api/submissions — Fetches a master list of all candidate submissions across the room.
PUT /api/session/start — Opens the testing session globally, starting the countdown clock.
PUT /api/session/close — Closes the testing session globally, freezing all candidate operations.
POST /api/submissions/{id}/recheck — Requests the automatic checking engine to test and re-grade a specific submission ID again.
PUT /api/results/{candidate_id}/confirm — Finalizes and locks the scores for a specific candidate ID.

Manager Module

GET /api/statistics/summary — Computes and returns aggregated manager summary metrics.
GET /api/statistics/ranking — Generates the real-time leaderboard data sorted by highest score.
GET /api/statistics/status — Compiles the absolute counts of total 'Pass' vs 'Fail' candidates.
GET /api/report — Compiles and delivers the final export payload/report data format.


Response Formats
The evaluation checking engine (Auto-grader) expects strict JSON formatting. It is entirely case-sensitive. Ensure all string keys match exactly.
Success Response Envelope (HTTP 200 / 201)
json{
  "success": true,
  "data": {},
  "meta": {}
}

Note: The meta object must always be explicitly defined in the payload root, even if it is left empty {}.

Error Response Envelope (HTTP 4xx / 500)
json{
  "success": false,
  "message": "Detailed and readable error message string describing the exact validation or server fault"
}

7. Business Rules
These rules dictate the core business logic layer. Failure to enforce these at the backend middleware/controller layer will result in immediate disqualification of the endpoint's score.

Single Active Submission Constraint: A candidate may have only one active submission for the current session.
Session Lifecycle Constraint: A candidate cannot create (POST) or update (PUT) any submission before the global session starts or after the global session closes.
Judge Session Control: Judge users are the only users allowed to start or close the session. All other roles must be blocked from /api/session/* mutations.
Manager Read-Only: Manager users are read-only users. They must be blocked from executing any state-changing HTTP methods (POST, PUT, DELETE).
Candidate Data Isolation: Candidates can access only their own submission and result data. Candidate A cannot read Candidate B's results.
LAN URL Validation: Submitted Frontend and Backend URLs must be syntactically valid internal LAN addresses and must be reachable by the testing network checking system.
State Blocking (HTTP 403): If the testing session is closed, protected operations (such as submitting or updating links) must intercept the request and return an HTTP 403 Forbidden error response immediately.
Seed Data Compatibility: The final application must continue to work with the provided seed data, although the database schema may be extended if required.
